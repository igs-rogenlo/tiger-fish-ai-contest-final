Harness Engineering — 實際內容範例

每個職能 Agent 背後的工程結構，取自實際檔案（路徑/帳號等已去識別化）。這是「Agent 工程」而非「prompt 工程」的佐證。

📋 AGENTS.md 規則檔slot-client · 約 300 行

每個 Agent 一份的標準作業規則，啟動即載入。以下為「輸出格式」段（節錄）：
## ⚠️ 輸出格式（最高優先）
使用者在手機上也會看，所以：
1. 結構化資料優先用清單，不要用 markdown 表格（手機欄位會跑掉）
2. 架構圖/流程圖/sequence/state/ER → 一律用 Mermaid，
   過 6-gate hardlock（含 vision check）再交付
3. 大量表格/報表/訊息 PNG → Pillow 產圖（venv + 中文字體）
4. 程式碼 → code block；檔案路徑 → 用 `file_path:line_number` 方便跳轉
5. 變更回報 → 固定三段：改了哪些檔案 / 新增哪些資源 / 有沒有碰共用模組
＊另含 Session 啟動序列、自驗證協議、per-user Git 等段落（敏感段不公開）。

📐 系統協議 docs/system/把「做完」寫成可機器驗證的契約

不是只放幾份說明檔，而是把「怎樣算做完」拆成 Agent 能自動逐項檢查的 gate。檔案群：SESSION_STARTUP · SELF_CHECK_PROTOCOL · DIAGRAM_HARDLOCK · FRESHNESS_CHECK · DELIVERY_RULES · ACP_PIPELINE …
以 DIAGRAM_HARDLOCK.md（交付架構／流程圖的硬規則）為例 —— 它定義「6 個 Gate 全 PASS 才能說完成」，每條都是「動作＋通過條件」兩段式（實際內容節錄）：
# │ Gate              │ 通過條件
2 │ mermaid_valid     │ mermaid parser 解析退出碼 0、無 syntax error
3 │ rendered_artifact │ 渲染檔存在、且 mtime 晚於 source（source 改過必重 render）
4 │ size_check        │ 渲染圖 ≥ 600×400px；不可全白；不可只有 title
6 │ vision_pass       │ vision LLM 對照 source 三項全 PASS：
  │                   │   READABLE（字可讀）· NO_OVERLAP（不重疊）
  │                   │   · SEMANTICALLY_CORRECT（節點數／方向對齊 source）
任一 Gate 沒過 → Agent 不可回「完成」，必須標記 FAIL、修補後重跑。「可機器驗證的契約」不是口號，是這種一條條能跑的判定。

🔧 TOOLS.md 工具路由 ＋ 參考專案對照相似類型直接複用

每個 Agent 預配好工具鏈與路徑（Python venv／中文字體／Cocos build wrapper／ocb 瀏覽器自動化），並維護一張參考專案對照表 —— 新機台依「類型」挑最相近的範本直接複用，不從零開始（Game ID／專案名已去識別化）：
## 參考專案清單（新機台依「類型」挑最近範本複用）
  機台 A   —  Cocos 3.8.8          —  三收模板
  機台 B   —  Cocos 3.8.8          —  掉落消去模板
  機台 C   —  Cocos 3.8.8          —  極速模板
  機台 D   —  Cocos 3.8.8          —  完整型骨架
  舊參考×2 —  Cocos 2.4.13／3.8.6  —  舊版參考

→ 例：新案是「掉落消去」型 → 直接複用機台 B 的骨架再改
＊另含 Cocos 雙路徑（互動走 MCP server、批次走 cocos-build wrapper，依專案 package.json 自動挑 Cocos 版本）、Pillow 產圖、慣用 grep pattern 等工具組合，開箱即用、統一維護。

🧠 知識庫（.md）＋ 記憶（向量庫）兩層互補

① 知識庫 ＝ 人／Agent 維護的 .md（可 git diff、可審）：裡面是踩過坑才寫得出的規則。例如 naming-conventions.md 這條（避免新機台撞到共用模組命名）：
## 不要在 Game/ 內使用的名稱
（共用模組的保留類別名，重複定義會污染命名空間、撞到所有遊戲共用模組）
   MachineHost · SlotGameMediator · Symbol · DropModule …
→ 要繼承就用帶前綴子類別：G{NN}_MachineHost extends MachineHost
② 記憶 ＝ LanceDB 向量庫（mlpro），不是 .md：Agent 被糾正／踩坑時自動寫入，每輪對話自動召回最相關的幾條 —— 這才是「對話即更新、越用越強」。實際 store 介面（節錄）：
memory_store({
  text, l0_abstract:"≤180", l1_overview:"≤500", l2_content:"≤1500",
  category,      // fact(更正/lesson) · decision · preference · entity
  importance,    // 0.5 預設 · 0.85+ lesson · 0.9+ critical
  scope:"agent:<id>"
})
# 每輪自動注入 <relevant-memories>；memory_recall({query,limit}) 可主動查
# 自檢協議：用戶更正/lesson → memory_store(category:"fact", importance:0.85+)
知識庫（.md，結構化參考、人可讀可審）＋ 記憶（LanceDB 向量、自動召回學習）兩層互補 ＝ 越用越強。另含 network-protocol、code-review-criteria、cocos-version-matrix、patterns/ 等 20+ 份知識。

✅ 自驗證 Gate（SELF_CHECK_PROTOCOL）交付前 AI 自己負責

交付前 Agent 自己跑契約檢查，不靠下游才發現問題。四個核心原則（節錄）：
🔒 契約優先：每類交付物配 HARDLOCK（圖/規格/code…），
   定義「什麼算完成」的可機器驗證 gate；條目用「動作＋通過條件」兩段式寫

🔒 自驗證循環：完成交付 → 跑契約 gates → 全 PASS？
   ├─ YES → 留證 → 交付
   └─ NO  → 進失敗處理（修交付物）

🔒 失敗不改標準：gate FAIL 不接受降標準/override；永遠修交付物（圖/spec/code），
   不修 gate；改契約須留 audit trail

🔒 可觀察留證：每次交付附通過證據
