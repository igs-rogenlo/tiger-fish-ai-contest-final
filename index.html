System Architecture

系統架構：完整技術架構

Multi-Node Architecture

多節點架構：大腦一台、手腳多台

Gateway「大腦」負責編排與服務；吃重或需特定環境的執行，外派到「執行節點」（機率＝獨立 Windows 機、美術＝同機的隔離角色）—— 算力隔離、平行不卡主腦、可水平擴充。

⚡ 算力隔離
機率快測動輒模擬數百萬次 spin，放獨立機器跑，不卡大腦服務其他人。
🧩 環境隔離
機率需 Windows 快測環境、美術需創意軟體環境 —— 各跑最適環境。
📈 水平擴充
手腳機器跑重活、大腦持續服務；要更多產能，加一台節點即可。
Multi-Channel Access

多通道接入：在你慣用的地方找 Agent

同仁不必換工具 —— Telegram、Discord、自架 Open WebUI 都能用自然語言找對應 Agent，且不同通道各有擅長場景。

💬 Discord
多人 × 多 Agent 議題討論：拉同仁＋對應 Agent 開議題（如「上週營收降低10%追蹤」），或拉多個 Agent 一起討論。
✈️ Telegram
排程提醒／系統訊息分析：叫 Agent 做排程提醒；群組自動收系統訊息、彙整提醒（活動提醒、大事件儀表板）。
🌐 Open WebUI（自架）
內部自架、資料不外流：架在公司內網，介面與對話紀錄留在內網；ChatGPT 式完整介面、免安裝，瀏覽器直接與各職能 Agent 對話、開議題。

Why GPT-5.5 as the Brain

為什麼選 GPT-5.5 當「主腦」

主腦要想清楚＋調對工具（一個任務常串十幾次工具）。選型不押「單一跑分第一」（各家排行榜互有勝負），而看兩個務實條件：能不能當 OpenClaw 後端、適不適合 agentic 任務：

✅ 官方支援當後端（決定因素）
GPT-5.5（Codex）是 OpenClaw 官方支援的後端，可直接、穩定當 Agent 主腦 —— 這是最關鍵的硬條件。
🧠 頂尖 agentic 模型
OpenAI 官方定位它「為 agentic 工作而生」；公開評測（Artificial Analysis）穩居第一梯隊（與 Opus 4.8 並列前段）。
⚡ 正好擅長主腦要的事
在終端／CLI 工作流、長上下文、抽象推論等任務領先 —— 主腦最吃這些（工具調用本身各家互有勝負）。
為什麼不是 Claude Opus 4.8 當主腦？ —— Opus 4.8 在多項評測也頂尖（甚至略前），但它的訂閱方案不能當 OpenClaw 後端（不支援） → 改用在程式／腳本（Claude Code，正好是它最強的領域）；主腦撞額度時備援自動切 Kiro Opus 4.6。

Memory Lifecycle

記憶會自己強化、淡忘、整理

記憶不是「只存不刪」的倉庫 —— 它像人腦：常用的留強、不用的淡出、相似的合併、跨日接得上。這套生命週期讓記憶庫越用越精、不會膨脹成垃圾場。

⏳ 時間衰減
每筆記憶會隨時間慢慢變淡、不常用的逐漸淡出；越重要的忘得越慢、幾乎不忘。
🔼 三層升降
分「核心／常用／邊緣」三層：常被用到的往上升、很久沒用的往下降、最後淘汰。
💪 存取強化
被手動查詢叫出來的會被加強、更不容易忘（自動帶出的不算，避免雜訊被誤強化）。
🧹 自動整理、不膨脹 —— 記憶庫每 30 分鐘自動整理（合併儲存碎片、清理舊版本）控制體積；並具備相似舊記憶的語意合併能力。
雲端模型負責「想」，這套記憶庫負責「記」 —— 像人腦一樣會強化、淡忘、整理，越用越精。

Defense in Depth

資安縱深：兩個關鍵風險面 —— 自動排程 × 記憶寫入

本平台內部部署、沒有對外公開服務；在多人使用下，我們特別防護的兩個面是 —— ① 使用者排的自動排程任務被注入或冒充（之後會自動執行）、② 惡意內容被寫進長期記憶持續影響。除了基本權限分級，我們對這兩面做縱深防護：

🛡 攔下被偷塞的惡意指令
排程任務先過注入偵測：本地 Qwen 先判，可疑或失效才再加判 Codex、取較嚴；兩者都失效就直接拒（fail-closed）。
🔏 防偽造、防冒名
排程任務用密鑰蓋「防偽封條」（HMAC，像支票防偽水印），被偷改就驗不過；身份只認平台給的真實 ID（像門禁刷卡），冒名也碰不到別人的排程。
🧹 寫進記憶前先篩毒
長期記憶寫入前做 secrets 遮罩（token／私鑰／路徑），機敏資料不會被寫進長期記憶庫。
🚦 限定發送對象＋用量上限 —— 排程任務若要把結果發到某 Telegram 群，群組須在核准名單、否則擋（防資料外送）；每人排程的數量與頻率有上限（防洗版灌爆）。
分類器、HMAC＋身份、發送白名單＋配額、記憶遮敏：使用時即時把關；新 Skill 上線前人工審核。
Reliability

可靠性：主腦掛了、撞額度、服務掛了，怎麼辦

不是靠人盯著 —— 設定即保證的高可用：自動偵測、自動切換、自動重啟，問題即時發現、不影響使用者。

🔄 主腦自動 fallback
主腦 GPT-5.5 掛掉 → 自動切備援 Kiro Opus 4.6（設定保證、非人工介入）。
🎫 多帳號自動輪替
撞額度（rate-limit／overload）→ 自動換下一個 OAuth 帳號；冷卻採 24h 滾動視窗、過後自動恢復。
💓 崩潰自動拉起
Gateway 由系統服務管理器（launchd）監看：崩潰自動拉起＋重啟節流（防 boot-loop）＋重啟交接狀態紀錄。
📟 額度監控告警
429 監控（已常駐排程）統計異常、超門檻 Telegram 告警，分辨「設計輪替 vs 真故障」。
把實證②的「看得到」延伸成「看得到＋掛了自動撐住」—— 服務掛掉自動拉回、主腦／額度出狀況自動切換；程式 bug 則由人操作 Agent 追蹤、判斷與修復。

Why Not Just a Personal Agent

和「自己電腦裝 AI Agent」差在哪

現在人人都能裝 Claude Code／Cursor。差別不在「誰的 Agent 聰明」，而是定位不同 —— 個人 Agent 讓「一個人」做得更快、把機敏資料留在本機；我們的框架放大的是「整個組織」—— 越做越快、知識留存、集中治理。不是取代，是各守其位。
